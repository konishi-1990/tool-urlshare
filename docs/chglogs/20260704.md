## 実施内容

### 1. 本番デプロイ環境の実装

設計書 `docs/deploy.md`（計画段階）に沿って各ファイルを作成・修正。

| ファイル | 区分 | 内容 |
|---|---|---|
| `.github/workflows/deploy-product.yml` | 新規 | PR マージ（`pull_request: closed` + `merged==true`, base=`product`）で発動。SSH で VPS の `deploy.sh` を実行。`workflow_dispatch`（手動）併設、`concurrency` で二重デプロイ防止 |
| `docker-compose.prod.yml` | 新規 | ポート非公開。上位 nginx-proxy に `VIRTUAL_HOST`/`LETSENCRYPT_HOST` で自動ルーティング。php=production・db 隔離・`restart: always` |
| `.dockerignore` | 新規 | production ビルド（context=repo root）から `vendor`/`node_modules`/`.git` 等を除外 |
| `script/deploy.sh` | 修正 | LMS 用テンプレを本プロジェクト用に修正（`product` / prod compose / サービス名 `php` / `/var/www/backend`）。composer/npm はビルド時実施のため実行時スキップ |
| `docs/deploy.md` | 修正 | ステータスを「実装済み」に更新 |

#### 要件との対応
- ✅ `main → product` の PR＋マージで発動（`branches: [product]` + `merged==true` ガード）
- ✅ ポート公開不要（`expose` のみ・`nginx-proxy` 外部ネットワーク参加、80/443 は上位プロキシが終端）

### 4. 初回デプロイ時のトラブル対応

#### 手動トリガー実行の権限エラー
- `gh workflow run` が HTTP 403（admin 権限不足）。gh CLI の認証アカウント（`kenichikonishi-boop`）が対象リポジトリに admin 権限を持たないため
- 対処: GitHub UI の「Run workflow」から手動実行、または admin 権限アカウントで `gh auth login` する運用を案内

#### `key:generate` が Read-only file system で失敗
- 原因: `docker-compose.prod.yml` で `backend/.env` を `:ro`（読み取り専用）マウントしているため、`key:generate --force` が `.env` に書き込めない（意図した仕様）
- 対処: `key:generate --show` で値だけ生成 → ホスト側 `backend/.env` に手動記入 → `config:clear && config:cache` で反映
- `docs/deploy.md` §8 の初回手順を `:ro` 前提の正しい手順に修正

#### 管理画面ログインユーザーの不在
- 調査結果: 本番 DB にユーザーが 0 件（`deploy.sh`・初回手順は `migrate` のみで `db:seed` を実行しない）
- `DatabaseSeeder` を実行しても `UserFactory` 既定が `is_admin=false` のため管理画面（`EnsureUserIsAdmin` 要求）にはログイン不可。`register` API も管理者を作れない
- 対処案内: `php artisan tinker` で `is_admin=true` のユーザーを作成（User モデルは `password => 'hashed'` キャストのため平文パスワードを渡す）
