# 作業ログ 2026-07-21

## 概要

本番（PRD）環境でログイン時に「エラーが発生しました。しばらく経ってからお試しください。」が表示されログインできない不具合を修正。原因は **ビルド時 `config:cache` による `DB_CONNECTION=sqlite` の焼き込み**。恒久対策として、env に依存するキャッシュ生成と migrate を「コンテナ起動時」に実行する `entrypoint.sh` を導入した。

---

## 不具合の原因

- 本番イメージの `Dockerfile`（production stage）が **ビルド時に `php artisan config:cache`** を実行していた。
- ビルド時は `.env` が不在のため、`config/database.php` の既定 `env('DB_CONNECTION', 'sqlite')` → **`sqlite`** が `bootstrap/cache/config.php` に焼き込まれる。
- ログイン API（`AuthController::login`）の `Auth::attempt()` が既定 `web` ガードで users を検索する際、キャッシュ済み設定が sqlite を指すため実行時の pgsql が無視され、存在しない sqlite を参照 → `SQLSTATE / no such table: users` → **HTTP 500**。
- フロント（`frontend/index.html`）は 401/422 以外を一律「エラーが発生しました。…」と表示するため、認証失敗ではなくサーバーエラーであることが埋もれていた。

### なぜ再発したか

- `script/deploy.sh` の手順4で起動後に `config:clear && config:cache` を再実行し pgsql で上書きしていたが、**これはデプロイ実行時の一度きり**。
- `restart: always` によるコンテナ再生成・VPS 再起動・docker 再起動でコンテナが作り直されると、イメージに焼き込まれた sqlite 設定へ戻り、手順4は再実行されない → ログイン 500。

---

## 実施内容

| ファイル | 区分 | 内容 |
|---|---|---|
| `docker/php/Dockerfile` | 修正 | production stage のビルド時 `config:cache` / `route:cache` / `view:cache` を削除。`entrypoint.sh` を配置し `ENTRYPOINT` に設定（`CMD ["php-fpm"]`） |
| `docker/php/entrypoint.sh` | 新規 | コンテナ**起動時**に、マウント済み `.env` を元に `config:clear && config:cache && route:cache && view:cache` を実行し、続けて `migrate --force`。最後に `exec "$@"`（php-fpm）へ |

これにより、コンテナが再生成・再起動されても毎回マウント済み本番 `.env`（pgsql）から正しく再キャッシュされ、sqlite への意図しないフォールバックが起きない。

---

## 復旧手順

### 即時ホットフィックス（再ビルド不要・任意）

```bash
cd <DEPLOY_DIR>
docker compose -f docker-compose.prod.yml exec -T php sh -lc \
  "cd /var/www/backend && php artisan config:clear && php artisan config:cache && php artisan migrate --force"
```

### 恒久対応

`main → product` の PR マージで自動デプロイ（`deploy.sh` が `up -d --build` で本修正入りイメージを再ビルド）。
